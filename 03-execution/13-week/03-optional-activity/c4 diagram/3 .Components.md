## ¿Por qué dejar solo `ValidationException`?

Se deja únicamente `ValidationException` porque es la excepción más utilizada en un sistema al momento de validar datos antes de guardarlos o procesarlos.

- Campos vacíos
- Correos inválidos

- Datos duplicados
- Longitud mínima o máxima

# Explicación de la arquitectura

Esta arquitectura es conocida como:

## Arquitectura Modular o Feature Based

En lugar de organizar el proyecto por capas globales (`Controller`, `Service`, `Repository` separados para todo el sistema), se organiza por funcionalidades o módulos.

Cada módulo tiene todo lo necesario para funcionar de manera independiente.

---

# ¿Por qué usar esta arquitectura?

Porque:

- Es más moderna
- Escala mejor
- Facilita mantenimiento
- Reduce dependencias entre módulos
- Hace más fácil trabajar en equipo
- Cada módulo tiene responsabilidad propia

---

# Explicación de `Modules`

## `Modules/Security`

Contiene toda la lógica relacionada con:

- Usuarios
- Roles
- Login
- JWT
- Autenticación
- Autorización

Por eso tiene:

- `Entity`
- `Repository`
- `Service`
- `Controller`
- `DTO`
- `Mapper`

Todo encapsulado dentro del mismo módulo.

---

## `Modules/Inventory`

Contiene toda la lógica relacionada con inventario.

Por ejemplo:

- Productos
- Stock
- Entradas
- Salidas
- Movimientos

Además incluye:

## `Validator`

Porque inventario normalmente necesita muchas validaciones:

- Stock negativo
- Productos duplicados
- Cantidades inválidas
- Límites de inventario

---

# Explicación de `Shared`

La carpeta `Shared` se deja porque contiene elementos reutilizables para TODO el proyecto.

---

## `Shared/Security`

Contiene elementos de seguridad globales:

- JWT Filter
- JWT Provider
- Configuración Spring Security
- PasswordEncoder
- AuthenticationEntryPoint

Se deja aquí porque todos los módulos pueden necesitar seguridad.

---

## `Shared/Config`

Configuraciones globales del proyecto.

Por ejemplo:

- Swagger
- CORS
- Beans
- ModelMapper
- OpenAPI
- Configuración de fechas
- Configuración de Jackson

Se centraliza para evitar repetir configuraciones.

---

## `Shared/Exceptions`

Manejo global de excepciones.

Por ejemplo:

- `ValidationException`
- `GlobalExceptionHandler`

Se deja aquí porque cualquier módulo puede lanzar excepciones.

---

## `Shared/Utils`

Utilidades reutilizables.

Por ejemplo:

- Generadores
- Helpers
- Formateadores
- Utilidades de fechas
- Utilidades JWT

Se comparte entre todos los módulos.

---

## `Shared/Constants`

Constantes globales del sistema.

Por ejemplo:

- Mensajes
- Roles
- Estados
- Límites
- Paths
- Variables estáticas

Ayuda a evitar texto repetido y mejora mantenimiento.

---

# ¿Por qué esta arquitectura es buena?

Porque:

- Cada módulo es independiente
- El código es más limpio
- Facilita escalar el sistema
- Facilita microservicios en el futuro
- Hace más fácil encontrar archivos
- Reduce acoplamiento
- Mejora organización profesional

Es una arquitectura muy utilizada actualmente en proyectos empresariales modernos con Spring Boot.

# DDD — Domain Driven Design

Es una arquitectura avanzada usada en sistemas empresariales grandes.

La idea principal es:

> El negocio es el centro del proyecto.

O sea:
la lógica del negocio se separa completamente de:

- la base de datos
- los controladores
- Spring Boot
- APIs
- frameworks

---

# ¿Por qué es difícil de entender al inicio?

Porque ya no se organiza como:

- Controller
- Service
- Repository

sino que ahora el proyecto se divide por responsabilidades reales del sistema.

DDD intenta representar el negocio como funciona en la vida real.

---

# Explicación sencilla por capas

## `Domain`
Es el corazón del sistema.

Aquí están:

- las reglas del negocio
- entidades principales
- validaciones importantes

Ejemplo:
cómo funciona un horario, inventario o usuario.

---

## `Application`
Organiza lo que el usuario quiere hacer.

Ejemplos:

- crear horario
- registrar inventario
- consultar disponibilidad

Coordina procesos del sistema.

---

## `Infrastructure`
Es la parte técnica.

Aquí va:

- Spring Boot
- JPA
- Base de datos
- JWT
- APIs externas

Hace que el sistema funcione técnicamente.

---

## `Presentation`
Es la entrada del sistema.

Aquí están:

- Controllers
- Endpoints REST
- Swagger
- GraphQL

Recibe las peticiones del usuario.

---

## `Shared`
Contiene cosas reutilizables para todo el proyecto.

Ejemplos:

- Exceptions
- Utils
- Validators
- Constants

Ayuda a no repetir código.

---

# Resumen rápido

| Capa | Función |
|---|---|
| Domain | Reglas del negocio |
| Application | Procesos y casos de uso |
| Infrastructure | Tecnología y base de datos |
| Presentation | Entrada del usuario |
| Shared | Código reutilizable |
